# Ruflo (claude-flow) + Agentic QE v3 ‚Äî Setup & Repair Kit

A six-script kit that gets [`ruflo`](https://www.npmjs.com/package/ruflo) (the renamed `claude-flow`) and the standalone [`agentic-qe`](https://www.npmjs.com/package/agentic-qe) plugin into a known-good state inside [Claude Code](https://github.com/anthropics/claude-code), and keeps them there session to session.

If you have ever seen one of these symptoms, this kit is for you:

- `agentdb_controllers` reports **15+ controllers disabled** even though ruflo says it started cleanly
- `memory_search` returns nothing on a project that should have entries
- The Claude Code status line is **stuck on a stub** ("‚ñä RuFlo V3.x" with no live data) after running `aqe init`
- A fresh `npx ruflo@latest ‚Ä¶` call **resolves a stale agentdb@1.x** from somewhere on disk
- After a Node.js / Claude Code / ruflo upgrade, MCP starts but the rich features (ReasoningBank, intelligence layer, swarm coordination) silently degrade

The scripts are **idempotent**, **dry-run-safe**, and **report what they change**. They are aimed at one box (the developer machine running Claude Code), not at production.

> **New to this kit?** Start with the section ["RuFlo vs Agentic QE ‚Äî what does what"](#ruflo-vs-agentic-qe--what-does-what) below for a quick mental model of which tool owns which surface.

## Scripts at a glance

Every script is now invoked through the single `bin/ruflo-kit` dispatcher, which takes a `<command>` and a positional `<target>` codebase path (defaults to `$(pwd)`). The implementations live in `lib/` (renamed from the old flat `scripts/` layout).

| Command (`bin/ruflo-kit ‚Ä¶`) | Implementation | Purpose | When to run |
|---|---|---|---|
| **`init <target>`** | `lib/init.sh` | One-shot, idempotent project initialization: prereq check ‚Üí `ruflo init` ‚Üí `ruflo memory init` ‚Üí `agentic-qe init` ‚Üí **`.claude` template backfill (Step 4.5)** ‚Üí `fix-ruflo` (AgentDB patches) ‚Üí `fix-statusbar` ‚Üí `fix-aqe` ‚Üí activation table ‚Üí seed memory.db ‚Üí 10-point verification. | First-time bootstrap of a new project, or after `--force` to wipe and re-init. |
| **`fix-ruflo <target>`** | `lib/fix-ruflo.sh` | 11-step diagnosis + repair of the ruflo MCP setup: patches `controller-registry.js`, removes shadow `agentdb`, fixes `.mcp.json`, re-registers MCP, cleans ghost state, updates `agentic-qe`. | First-time setup, after any Node/ruflo upgrade, whenever ReasoningBank or memory look broken. |
| **`fix-statusbar <target>`** | `lib/fix-statusbar.sh` | Restores the rich Claude Code status line that shows live ruflo + Agentic QE v3 stats. Survives `aqe init` clobbering. | Status line shows a stub, missing emoji, or no live numbers. |
| **`session <target>`** | `lib/session-init.sh` | Runs `fix-ruflo` + `fix-statusbar` in quiet mode, then verifies MCP, persistent storage, AgentDB controllers, native RuVector binaries. | At the start of every Claude Code session for full effective use. |
| **`upgrade <target>`** | `lib/upgrade.sh` | One-shot Phase-A upgrade: stops daemon, bumps global ruflo, wipes `~/.npm/_npx`, re-hydrates, re-runs `fix-ruflo`, then re-runs the activation layer via `init <target> --reactivate`. | When migrating across a major ruflo bump (e.g. 3.6 ‚Üí 3.10). |
| **`health <target>`** | `lib/health.sh` | Growth-delta monitor. Captures the same metrics ruflo's statusbar shows (memory entries, SONA patterns/trajectories, MoE accuracy, ReasoningBank patterns, DB rows/sizes, daemon status, hive workers); compares against `.claude-flow/data/health-last.json` snapshot from the previous run; prints color-coded deltas; flatlines >24h warn; regressions exit non-zero (CI-friendly). | After any session where you want to confirm the loop is actually learning; in cron/CI to gate on growth. |

`init` and `session` orchestrate the others; the fix scripts are standalone. The init command handles **bootstrap** (calling `ruflo init` / `aqe init` for the first time); the session command handles **per-session maintenance** (assumes the project is already initialized). The dispatcher resolves `KIT_DIR` (this clone) vs `TARGET_DIR` (the codebase path you pass) via the shared `lib/common.sh`.

## RuFlo vs Agentic QE ‚Äî what does what

The kit installs three separate npm packages that cooperate but own different surfaces. Confusion about which package owns what is the single most common source of "I thought this was broken" issues ‚Äî the statusbar in particular splits its rows by domain (Tier 5), so reading the right chip for the right question matters.

**The three packages:**

| Package | Role | Owns | Database |
|---|---|---|---|
| `ruflo` (npm: `ruflo`) | MCP orchestration platform ‚Äî agents, swarms, hive-mind, hooks, neural training, daemon | `.claude-flow/` config + state, MCP server registration, hooks runtime, swarm-state, hive-mind, neural patterns | `.swarm/memory.db` (sql.js + HNSW) |
| `agentic-qe` (npm: `agentic-qe`) | Quality engineering plugin ‚Äî 60 specialist agents, 13 bounded contexts, SONA learning, MoE routing, ReasoningBank | `.agentic-qe/` config + state, ~115 `.claude/skills/`, 60 `.claude/agents/`, ~20 `aqe-*` commands, the "Agentic QE v3" section in `CLAUDE.md` | `.agentic-qe/memory.db` (sqlite + embeddings) |
| `agentdb` (npm: `agentdb`, pinned to `@3.0.0-alpha.10` ‚Äî pre-pivot; do NOT bump to alpha.12+, see Patch 17.1) | Direct AgentDB primitives ‚Äî attention, reflexion, skills, causal, learning-session tools | Project-scope MCP server entry (Tier 6.1); shares `.swarm/memory.db` with ruflo via @claude-flow/memory's `ReasoningBank` controller | (no separate DB ‚Äî uses `.swarm/memory.db`) |

**MCP server layout in `.mcp.json`:**

```json
{
  "mcpServers": {
    "claude-flow":  { "command": "ruflo", "args": ["mcp", "start"], "autoStart": true },  // global bin, NOT `npx -y ruflo@latest` ‚Äî see Patch 18
    "agentic-qe":   { "command": "aqe-mcp", "args": [] },  // live config uses the global `aqe-mcp` bin (NOT npx) ‚Äî see Patch 22.3
    "agentdb":      { "command": "npx", "args": ["-y", "agentdb@3.0.0-alpha.10",  "mcp", "start"] }
  }
}
```

After `bin/ruflo-kit fix-ruflo <target>`, expect three servers (one auto-start, two opt-in via `.claude/settings.json`'s `enabledMcpjsonServers`). MCP chip rendering rule (Patch 9): chip total = deduped union of user-scope + project-scope servers; enabled = user-scope (always on) + (project-scope ‚à© enabledMcpjsonServers).

**Statusbar row ‚Üí package mapping:**

| Statusbar row | Package | Counter source |
|---|---|---|
| `ü§ñ Swarm [N/15]` | ruflo | `swarm-state.json` agents (ruflo's hive-mind workers) |
| `üë• Sub N` | Claude Code (NOT ruflo, NOT AQE) | `~/.claude/projects/<key>/<session>/subagents/*.jsonl` ‚Äî counts your `Agent({ run_in_background: true })` calls |
| `ü™ù N/M` | ruflo | `M = total hooks defined by ruflo` (currently 21 in the latest ruflo template ‚Äî increases as ruflo adds new hook types). `N = hooks actively wired in `.claude/settings.json`'s `hooks` block`. When all defined hooks are wired, this renders `21/21` green. The "active vs defined" semantic was previously ambiguous; **clarification (Tier 6.2)**: this is `wired/defined`, not `pre-task active/scheduled`. |
| `üíæ NMB` (Swarm row) | kit-total | All three DBs + hnsw.index ‚Äî cross-domain summary |
| `üßÝ N%` | ruflo | Real MoE **routing accuracy** from `ruflo hooks intelligence stats` (Patch 20.7 `RUFLO-INTEL-V1`). Was a `.swarm/memory.db` file-size proxy; pre-3.10.x BRAIN-INPUT-TOTAL-V1 no longer applies |
| `üì∂ SONA` | ruflo | SONA learning ladder: trajectories-driven 5-tier `[‚óè‚óè‚óè‚óã‚óã]` + patterns + quality, from `hooks intelligence stats` / `neural/stats.json` (Patch 20.7) |

| `üß© Plugins row` | ruflo (plugin scan) | `node_modules/@claude-flow/plugin-*` packages |

**Rule of thumb:** if a chip changes on the AgentDB row, ruflo's swarm/memory layer changed. If a chip changes on the AQE footer, an AQE agent did something (captured an experience, scored a pattern, advanced a trajectory). Cross-domain summary chips (`üíæ MB`, `üßÝ %`) live on the universal `ü§ñ Swarm` row.

**Which script touches what** (read this when a script logs something surprising):

| Script | Touches ruflo state? | Touches AQE state? | Touches Claude Code? | Touches npx cache? |
|---|---|---|---|---|
| `init-ruflo-aqe-agentdb.sh` | yes (init + activation) | yes (init) | no | no |

If you only remember one thing: `fix-ruflo.sh` is for the MCP and AgentDB/ReasoningBank substrate; `fix-statusbar.sh` is for what you can SEE; `ruflo-health.sh` answers "is the loop actually growing"; `init-...` is the one-shot bootstrap; `ruflo-session-init.sh` is the daily entry; `upgrade-...` is when ruflo itself bumps a major version.

## Prerequisites

- macOS (Darwin) or Linux. Tested on macOS arm64.
- Node.js ‚â• 18 (recommended ‚â• 22).
- `npm` reachable on the public registry, OR a local mirror.
- Claude Code CLI (`claude`) installed and on `PATH`. Optional ‚Äî without it the MCP-registration step is skipped.
- Bash (these are `#!/usr/bin/env bash`, not POSIX `sh`).

The scripts touch only **user-scope** locations: `~/.npm/_npx`, `~/node_modules`, the project's `.mcp.json`, `.claude/`, `.claude-flow/`, `node_modules/`. They do **not** require `sudo`.

## Quick start

Clone the kit once, then point it at the codebase you want to set up. The kit (this clone) is `KIT_DIR`; the codebase path you pass is `TARGET_DIR` (defaults to `$(pwd)`):

```bash
# one time per machine ‚Äî clone the kit
git clone <repo> ruflo-aqe-kit && cd ruflo-aqe-kit

# one time per project (bootstrap) ‚Äî pass the target codebase path
bin/ruflo-kit init /path/to/codebase

# every Claude Code session afterwards
bin/ruflo-kit session /path/to/codebase
```

The bootstrap step is idempotent: it skips `ruflo init` / `aqe init` if it detects they have already been run, and only re-applies the patches. Pass `--force` to wipe state and re-init from scratch, or `--dry-run` to preview without changes.

That's the whole happy path.

If something looks wrong (controllers disabled, memory empty, status line dead), run:

```bash
bin/ruflo-kit fix-ruflo /path/to/codebase       # patches
bin/ruflo-kit fix-statusbar /path/to/codebase   # status line
# then restart Claude Code
```

A safe preview without changes:

```bash
bin/ruflo-kit init /path/to/codebase --dry-run
bin/ruflo-kit fix-ruflo /path/to/codebase --dry-run
bin/ruflo-kit upgrade /path/to/codebase --dry-run
```

## `init-ruflo-aqe-agentdb.sh` ‚Äî first-time project bootstrap

A single entry point that brings a fresh project from zero to a fully working ruflo + AQE + AgentDB setup with the **self-learning + self-improving loop activated**. Each of its steps (1, 2, 3, 4, **4.5**, 5, 6, 6.5, 7, 7.5, 8, 9) is gated by an on-disk or `ruflo`-stat check so the script can be re-run safely.

| Step | Section | What it does | Skip condition |
|---|---|---|---|
| 1/9 | Prerequisites | Verifies `node`, `npm`, `ruflo` (or falls back to `npx -y ruflo@latest`), `aqe` (or falls back to `npx -y agentic-qe@latest`), and `claude`. Aborts if `node` or `npm` are missing; warns and recommends `npm install -g agentic-qe` if `aqe` isn't on `PATH`. | Never skipped. |
| 2/9 | `ruflo init` | Creates `.claude/` (skills, commands, agents, helpers, settings.json with 7 hook types), `.claude-flow/` runtime, `.mcp.json` registering the `claude-flow` MCP server. | Skipped if `.claude-flow/config.yaml` + `.mcp.json` + `.claude/skills/` already exist. |
| 3/9 | `ruflo memory init` | Creates `.swarm/memory.db` (sql.js / WASM SQLite) with vector embeddings (`Xenova/all-MiniLM-L6-v2`, 384-dim), HNSW indexing, pattern learning, temporal decay, and the AgentDB controller registry. | Skipped if `.swarm/memory.db` already exists. |
| 4/9 | `agentic-qe init` | Runs `aqe init --auto` (the canonical first-time flag per the [agentic-qe README](https://github.com/proffesor-for-testing/agentic-qe#quick-start)). This both **initializes the SDK** (`.agentic-qe/memory.db`, PersistentSONAEngine, DreamScheduler, Queen Coordinator, 14 domains) **and copies the project-level integration**: ~115 skills, AQE agents, AQE commands, plus an "Agentic QE v3" section appended to `CLAUDE.md` with policies for Integrity, Test Execution, and Data Protection. Prefers the globally-installed `aqe` binary; falls back to `npx -y agentic-qe@latest` if missing. Uses `--upgrade` when `--force` is passed. | Skip-check requires **both** `.agentic-qe/memory.db` **and** the AQE-specific marker `.claude/skills/agentic-quality-engineering`. If the DB exists but the marker doesn't, the script identifies a **half-init** (SDK ran but `.claude/` templates were never copied ‚Äî happens with `--memory hybrid` or any non-`--auto` invocation) and re-runs `aqe init --auto --upgrade` automatically to repair it. |
| 4.5/9 | **.claude template backfill** (NEW, Patch 26) | Non-destructively backfills every agent/command/skill present in the global ruflo CLI template or the agentic-qe template but MISSING from `.claude/` (only ADDS, never overwrites). Fixes the AQE-first partial init (79/89 ruflo agents ‚Üí full 23 categories). | Skipped under `--reactivate` (without `--force`). Adds nothing once `.claude/` is complete. |
| 5/9 | AgentDB patches | Shells out to `lib/fix-ruflo.sh` ‚Äî applies the three `controller-registry.js` patches (ESM `path`, relative `agentdb` import, embedder argument) so ReasoningBank initializes correctly. | Never skipped (the patches detect already-applied state internally). |
| 6/9 | Statusbar restore | Shells out to `lib/fix-statusbar.sh` ‚Äî restores the rich statusline that `aqe init` clobbered in step 4. | Never skipped. |
| 7/9 | **Activation table** (NEW) | Runs the 7 sub-steps that take the project from "files on disk" to "self-learning loop on". Each sub-step is independently idempotent and respects `--dry-run` / `--force`. See the dedicated subsection below for details. | Each sub-step has its own skip rule. The roll-up reports `<N> run, <M> skipped` out of 7. |
| 8/9 | Seed memory | Stores a single `init` entry in `.swarm/memory.db` so subsequent `memory_search` calls aren't empty. Pre-checks via `ruflo memory retrieve -k init`; if the key already exists, skips immediately (the SQLite-backed memory enforces `UNIQUE(namespace, key)`, so a blind re-store would always fail). If the key is missing, attempts to store with exponential-backoff retry (1s ‚Üí 2s ‚Üí 4s ‚Üí 8s, 5 attempts) and a race-check between retries that treats a concurrent insert as success. | Skipped under `--dry-run`; also skipped (counted under "Steps skipped") on every re-run after the first successful seed. |
| 9/9 | Verification | 10-point check: `.mcp.json`, `.claude/settings.json`, `.claude/skills/`, `.claude-flow/config.yaml`, `.swarm/memory.db`, `.agentic-qe/memory.db`, **`.claude/skills/agentic-quality-engineering` (AQE half-init guard)**, `.claude/helpers/statusline.cjs`, **`.claude/agents` (ruflo full set backfilled ‚Äî Patch 26 guard)**, and `ruflo mcp status` reachability. | Never skipped. |

#### Step 7/9 ‚Äî Activation table sub-steps

The activation table is what turns a freshly-`init`-ed project (all-zero statusline) into a project where the self-learning + self-improving loop is actually running. It is the difference between "ruflo is installed" and "ruflo is alive".

| Sub-step | Command | What it produces | Skip condition |
|---|---|---|---|
| 7A | `ruflo swarm init --topology hierarchical-mesh --max-agents 15` | `.claude-flow/swarm/swarm-state.json` (coordinator + queue) | Skipped if `swarm-state.json` exists and mtime < 30 min |
| 7B | `ruflo embeddings init` | ONNX embeddings (`Xenova/all-MiniLM-L6-v2`, 384-dim) downloaded and registered | Skipped if `ruflo embeddings status` reports a `Provider:` line |
| 7C | `ruflo memory store -k init/hnsw-seed-{1,2,3} -v bootstrap` (√ó3 idempotent seeds) | Triggers the lazy HNSW index build (HNSW lives in-DB, not on filesystem ‚Äî confirmed empirically) | Skipped if either `.swarm/hnsw.index` is non-empty **OR** `ruflo memory stats` reports `HNSW Index ‚îÇ active` (handles both ASCII `\|` and box-drawing `‚îÇ` separators) |
| 7D | `ruflo neural train -p coordination -e 50 --wasm --flash --contrastive` | SONA self-learning kicks off, seeds the pattern store and trajectory log | Skipped if `ruflo neural status` reports `Patterns: N` (>0) or `N patterns stored` (handles both number-after-noun and number-before-noun output formats) |
| 7E | `ruflo hooks pretrain -p . --depth shallow --with-embeddings` (depth overridable via `HOOKS_PRETRAIN_DEPTH`) | Pre-populates the hook routing pattern store so SendMessage / agent-spawn decisions have priors | Skipped if `ruflo hooks intelligence stats` reports `Patterns Learned ‚îÇ N` (>0) ‚Äî matches the box-drawing tabular output |
| 7F | `ruflo hive-mind init -t hierarchical-mesh -c byzantine -m 15 -p --memory-backend hybrid` | `.claude-flow/hive-mind/state.json` (collective memory + byzantine consensus + 15-agent capacity) | Skipped if `.claude-flow/hive-mind/state.json` exists |
| 7G | `ruflo daemon start` | The 5‚Äì12 background workers (audit, optimize, testgaps, map, document, etc.) that actually grow the system between user interactions | Skipped if `ruflo daemon status` matches `Status: ‚óè RUNNING` (U+25CF filled circle, with `Status:.*RUNNING` regex fallback). On `--force` and currently RUNNING, the script stops the daemon first then re-seats it. |

**Order matters**: A ‚Üí B ‚Üí C ‚Üí D ‚Üí E ‚Üí F ‚Üí G. `embeddings init` (7B) must precede any HNSW-triggering store (7C). `neural train` (7D) and `hooks pretrain` (7E) require the embedding provider from 7B. `daemon start` (7G) must come last; otherwise the workers crash-loop on uninitialised swarm/hive-mind state.

**Fail-soft**: any sub-step that fails logs to `/tmp/ruflo-*-{init,build,start}.log` and the script continues. Partial activation is more useful than aborted activation; you can repair individual sub-steps from the log without losing the rest. The summary surfaces partial state.

**Why the script doesn't trust `$?` for daemon status**: `ruflo daemon status` always exits 0 regardless of whether the daemon is running or stopped. The status check uses byte-literal `grep` for `Status: ‚óè RUNNING`.

### Flags

- `--force` ‚Äî re-runs each step even if its skip condition is true (passes `--force` to `ruflo init` and `--upgrade` to `aqe init`). Implies `--reactivate` (superset).
- `--reactivate` ‚Äî **lighter-touch refresh** (Tier 3). Skips steps 2/3/4 (`ruflo init`, `ruflo memory init`, `aqe init` ‚Äî the expensive ones that take minutes), then runs steps 5/6/7/8/9 (`fix-ruflo.sh`, `fix-statusbar.sh`, the activation table with daemon re-seat, seed memory, verification). Use when the bootstrap was already done but something in the activation layer stopped working (daemon died, swarm aged out, embeddings provider unseated, etc.). `--force` and `--reactivate` are mutually-exclusive at the banner level ‚Äî if both are passed, `--force` wins (banner shows only `--force`, and everything runs). The step 7G daemon re-seat fires under either flag.
- `--dry-run` ‚Äî prints every action that would run, without modifying anything. Orthogonal to `--force` / `--reactivate`.
- `-h` / `--help` ‚Äî shows the header comment.

### Exit codes

- `0` ‚Äî all 8 steps either ran successfully or were skipped because already complete.
- `1` ‚Äî at least one step reported an error (verification will show which).

### Typical first-run output

After a clean run on a fresh project you should see something like:

```text
Steps run:     8           # the 6 main steps + the seed + 7 activation sub-steps each counting
Steps skipped: 1           # only the seed when key already exists
Activation table: 7 run, 0 skipped
Verification:  10/10 checks passed
All systems ready.
```

After a re-run with no `--force`, every step should be skipped:

```text
Steps run:     2           # the two fix-* scripts, which are always idempotent and always run
Steps skipped: 11          # ruflo init, memory init, aqe init, the seed step, and all 7 activation sub-steps
Activation table: 0 run, 7 skipped
Verification:  9/9 checks passed
```

The "Activation table: 0 run, 7 skipped" line is the key idempotency signal ‚Äî if any of 7A‚Äì7G shows as "run" on a re-run, the skip-check for that sub-step is broken (was the source of a previous bug ‚Äî see "Local patches" item 7 if it regresses).

## `fix-ruflo.sh` ‚Äî what each step does

The script prints `[N/11] Section` headers as it runs. Knowing what each step is for makes it easier to interpret the output and decide whether warnings matter.

| Step | Section | What it does | Why it matters |
|---|---|---|---|
| 1/11 | Environment | Locates `node` and `ruflo`. Aborts if Node is missing. | Establishes a baseline. |
| 2/11 | Stale global packages | Removes a stale `claude-flow` global (since renamed to `ruflo`); installs or updates `ruflo@latest` globally. | Two globals with overlapping bin entries silently shadow each other. |
| 3/11 | Shadow `agentdb` detection | Walks `~/node_modules`, the project's `node_modules`, and every `~/.npm/_npx/*/node_modules/agentdb`. Removes top-level orphans whose major version is `< 3` (bundled deps under `@claude-flow/*/node_modules/` are kept). | A v1/v2 `agentdb` anywhere up the resolution chain resolves bare `import('agentdb')` to the wrong package ‚Äî disables every controller. |
| 4/11 | Local ruflo dependency check | Removes any local `package.json` dependency on `ruflo`, `@claude-flow/cli`, or `@claude-flow/memory`. | Local deps override the MCP-managed runtime and cause version drift. |

## `fix-statusbar.sh` ‚Äî what it does

After `aqe init` (or any tool that resets `.claude/helpers/`), the rich Claude Code status line is often replaced by a minimal stub. This script restores the dual ruflo + Agentic QE v3 display in four steps:

Before any patching, the script **detects the installed ruflo version dynamically** by running `ruflo --version` (falling back to `npx -y ruflo@latest --version`) and extracting the `<major>.<minor>(.<patch>)?` from the output. It then exposes two values to the node-side patcher: `RUFLO_VERSION_LABEL` (e.g. `V3.10`, used for any hardcoded header strings in older templates) and `RUFLO_VERSION_FULL` (e.g. `3.10.3`, used to rewrite the runtime-fallback default). If detection fails completely it defaults to `V3` / `3` so the script doesn't crash on offline machines.

1. **Patches `.claude/helpers/statusline.cjs`**:
   - **(1a)** Bumps any literal `‚ñä RuFlo V<X>[.<Y>] ` header to the detected label (no-op on modern templates, which concatenate the version at runtime).
   - **(1b)** Rewrites the hardcoded runtime fallback default `let pkgVersion = '3.X';` (which goes stale across ruflo releases) to the detected `<major>.<minor>.<patch>`. This is the patch that actually matters on the current template ‚Äî it ensures the rendered header shows the right number when ruflo isn't installed in any of the four `package.json` lookup paths the renderer checks.
   - **(1c)** Injects a `getAQEStats()` helper that reads `.agentic-qe/memory.db` via the `sqlite3` CLI (avoiding `better-sqlite3` ABI drift), and inserts a "Line 6: Agentic QE v3 footer" render block.
   - **(1d) ‚Äî `node --check` post-write guard.** Before any patching, the script snapshots `statusline.cjs` to `.bak`. After writing, it runs `node --check "$STATUSLINE_FILE"`; on syntax-error failure it restores from the `.bak` so the file is never left in a broken state. On success the `.bak` is removed. Note: the `.bak` is only useful if the source was clean when the snapshot was taken ‚Äî pre-existing corruption is repaired by step 1e's auto-repair branch (the `///g` case) rather than via `.bak` rollback. (Known limitation: arbitrary unknown corruption introduced outside the script's vocabulary will leave `.bak` containing the same corruption ‚Äî defense-in-depth, not a complete recovery mechanism.)
   - **(1e) ‚Äî stdin-header verify-and-guard.** Modern ruflo `statusline.cjs` ships with stdin renderers for model / cost / context-% / session-uptime (these are what populate the `Opus 4.7 ‚îÇ ‚è± 46m25s ‚îÇ ‚óè 32% ctx ‚îÇ $21.97` segment of the header). This step greps for **four anchor patterns** in `statusline.cjs`: `getStdinData()`, `data.model.display_name`, `data.cost.total_cost_usd`, `data.context_window.used_percentage`. If all four are present, logs `pass: stdin header fields wired (4/4)` and skips. If any are missing, logs a warning naming the missing anchors and tells the user to re-run `npx ruflo init --force` ‚Äî it does **not** auto-inject a renderer, because a future ruflo upgrade may have customised the file in ways the patcher doesn't understand, and warning loudly is safer than writing broken code.
   - **(1f) ‚Äî auto-repair** Detects a known-bad pattern from prior gist versions: `CWD.replace(///g, '-')` (where the slash-escaping was double-stripped by bash + JS template literal and `//` ended up parsed as a single-line comment, crashing the file with `SyntaxError: Unexpected token 'const'`). If found, repairs it to `CWD.replace(/\\//g, '-')`. The upstream `fix-ruflo.sh` step that originally introduced this corruption has been patched locally too (see "Local patches").
   - **(1g) ‚Äî MCP counter rewrite (sentinel `MCP-COUNT-PATCH-V1`)** Rewrites `getIntegrationStatus()` in `statusline.cjs` so the `MCP <N>/<M>` chip combines two distinct server scopes correctly. User-scope servers in `settings.json` `mcpServers` are always counted as enabled. Project-scope servers in `.mcp.json` are filtered through `settings.json` `enabledMcpjsonServers` (the Claude Code allowlist that only applies to the `.mcp.json` set, never to the user-scope set). `total` = deduped union of both scopes; `enabled` = user-scope count + filtered project-scope count. Before this patch, `enabledMcpjsonServers` was being applied to the wrong server set ‚Äî on a project with `mcpServers: {ruflo}` and `enabledMcpjsonServers: ['agentic-qe']`, the filter produced an empty intersection and the chip rendered `‚óè0/1` red despite the MCP server being healthy. The patch is gated by the existing 1d `node --check` + `.bak` rollback guard, so any future syntax-breaking edit auto-restores.
   - **(1h) ‚Äî Sqlite-aware Vectors + HNSW probe (sentinel `VECTORS-HNSW-PATCH-V1`)** Rewrites `getAgentDBStats()` in `statusline.cjs` to count real vector-bearing rows via `sqlite3 -readonly` probes of `.swarm/memory.db` (ruflo: sums `memory_entries WHERE embedding IS NOT NULL` + `pattern_embeddings` + `learning_state_embeddings`) and `.agentic-qe/memory.db` (AQE: sums `embeddings` + `qe_pattern_embeddings`). Treats a non-empty `vector_indexes` table as the in-DB HNSW-active signal ‚Äî required because HNSW indexes are stored as DB rows, not as a `.swarm/hnsw.index` file (lazy in-DB build; see Local Patch 7's 7C skip-rule for the same pattern). The HNSW-chip on the DDD-domains line then renders `‚ö° HNSW <N>x` from a vectorCount-keyed heuristic ladder (`<1000 ‚Üí 10x`, `‚â•1000 ‚Üí 150x`, `‚â•10000 ‚Üí 12500x`), replacing the prior misleading `üìö <N> patterns` chip whose count was an SQLite page-count artefact rather than a real pattern count. **`-readonly` mode is load-bearing** ‚Äî concurrent statusline renders during a live DB write would otherwise risk corruption. The patch reuses `safeStat` / `safeExec` helpers already in `statusline.cjs`; idempotent via the sentinel comment.
   - **(1i) ‚Äî Subagent file-scan counter (sentinel `SUBAGENTS-SCAN-V2`)** Rewrites the `// Sub-agents:` block in `statusline.cjs` (originally written by `fix-ruflo.sh` step 11d as a sentinel-less V1) to fix two compounding bugs that left `Sub` stuck at 0. (1) The project-key derivation `CWD.replace(/\//g, '-')` only mangled `/`, but Claude Code's `~/.claude/projects/<key>/` directory mangles **both** `/` and `_` to `-` ‚Äî projects with underscores in their path (e.g. `THE_AI`) silently missed their CC directory. New regex is `CWD.replace(/[\/_]/g, '-')`. (2) The real jsonl layout is `<projDir>/<key>/<sessionUUID>/subagents/*.jsonl`, not `<projDir>/<key>/subagents/*.jsonl` ‚Äî old code looked one level too shallow. New code recurses into per-session directories. The substring filter (`d.includes(projKey) || projKey.includes(d)`) is also tightened to exact equality (`d === projKey`) ‚Äî post-regex-fix the key is exact, so substring would only allow cross-project leakage. 10-minute mtime threshold preserved (right liveness signal). Anchored on the existing `// Sub-agents: prefer Claude Code stdin data, fallback to file scan` comment and the trailing `return { memoryMB, contextPct, intelligencePct, subAgents };`; protected by the existing step 1d `.bak` + `node --check` guard.

#### What the swarm-row chips count

The "Swarm" line on the statusline shows two independent counters:

- **`[N/15]`** (the `ü§ñ Swarm  ‚óØ [N/15]` chip) ‚Äî counts **ruflo-domain agents**, those spawned via `ruflo agent_spawn`, `mcp__claude-flow__swarm_init` with an `agents: [...]` list, or `ruflo swarm spawn`. This is the hive-mind / coordination worker count, capped at `CONFIG.maxAgents` (default 15).
- **`Sub N`** (the `üë• Sub N` chip) ‚Äî counts **Claude Code's `Agent`-tool background subagents**, i.e. anything spawned via `Agent({ run_in_background: true })` from inside Claude Code. The source of truth is the per-session `~/.claude/projects/<key>/<sessionUUID>/subagents/*.jsonl` files with `mtime < 10 min`.

These are independent. Running Claude Code `Agent` calls does NOT increment `[N/15]`, and running `ruflo agent_spawn` does NOT increment `Sub N`. If you've launched a coder/tester/reviewer pipeline via `Agent({ run_in_background: true })`, watch `Sub N`. If you're driving ruflo's hive-mind workers, watch `[N/15]`.

2. **Writes a dual fallback `.claude/helpers/statusline-v3.cjs`** ‚Äî a self-contained single-line renderer that reads both `.swarm/memory.db` (ruflo) and `.agentic-qe/memory.db` (Agentic QE) directly via `sqlite3`. Survives any future `aqe init` because Claude Code's `statusLine.command` falls back to this if `statusline.cjs` crashes.
3. **Patches `.claude/settings.json`** ‚Äî sets `statusLine.command` to `node statusline.cjs 2>/dev/null || node statusline-v3.cjs 2>/dev/null || echo "‚ñä RuFlo + Agentic QE v3"`.
4. **Removes the legacy `@claude-flow/plugin-agentic-qe`** if it is in `package.json` or `node_modules` (it has been superseded by the standalone `agentic-qe` package).

The script ends by rendering the status line to stdout so you can see the result without restarting Claude Code. If the rich render fails, it falls back to the dual stub and warns ‚Äî that means `statusline.cjs` itself has an unrelated issue; check `node .claude/helpers/statusline.cjs` directly to see the stack trace.

Restart Claude Code (Ctrl+C, `claude`) for the new status line to appear in the chrome.

## `ruflo-session-init.sh` ‚Äî the daily entry point

Seven-step verification that wraps `fix-ruflo.sh` + `fix-statusbar.sh` and adds runtime checks:

1. **Applies ruflo patches** by calling `fix-ruflo.sh` (silent ‚Äî only counts the ‚úì checks).
2. **Repairs the status bar** by calling `fix-statusbar.sh` (silent).
3. **Confirms the MCP server is running** by looking for `ruflo mcp start` in `ps aux`.
4. **Daemon auto-start** (NEW). Greps `ruflo daemon status` for `Status: ‚óè RUNNING`. If the daemon died between sessions (the common case ‚Äî daemons don't survive a machine reboot or a Claude Code restart), runs `ruflo daemon start` and logs to `/tmp/ruflo-session-daemon-start.log`. Idempotent ‚Äî a second run reports "daemon already running". This single step closed the most common cause of the loop silently stopping between sessions: the bootstrap left the daemon RUNNING, but the next time the user opens Claude Code, no one had restarted it.
5. **Inspects persistent storage** ‚Äî sizes of `.swarm/memory.db`, `.swarm/hnsw.index`, `.claude-flow/data/graph-state.json`, `.claude-flow/data/intelligence-snapshot.json`.
6. **Verifies AgentDB controllers** ‚Äî re-checks that the three `controller-registry.js` patches are still in place (catches the case where a cache wipe undid them).
7. **Checks RuVector native binaries** ‚Äî `sona.<arch>.node`, `attention.<arch>.node`, `ruvector-graph*.node` for the host's architecture (`darwin-arm64`, `darwin-x64`, etc.).

Final summary lists how many issues were found, and ends with the three MCP queries to run inside Claude Code to confirm `agentdb_controllers` is healthy, `memory_search` returns project data, and the intelligence layer is reachable.

**Recommended habit:** add the line below to your project's CLAUDE.md or session bootstrap notes:

```text
At the start of each session, run:  bin/ruflo-kit session <target>
```

## `upgrade-2026-05-26.sh` ‚Äî Phase-A major-version migration

A one-shot script for the ruflo 3.6.x ‚Üí 3.10.x bump (the version numbers are illustrative ‚Äî the script reads `npm view ruflo version` at runtime). It is **not idempotent** in the sense that it deletes the whole `~/.npm/_npx` cache; running it twice in a row is harmless but wasteful.

Sequence (8 steps, renumbered in Tier 3 to bracket the wipe with daemon lifecycle):

1. **Pre-flight** ‚Äî Node version, current vs latest ruflo, warn if `ruflo mcp start` is currently running.
2. **Daemon stop** (Tier 3, NEW) ‚Äî `ruflo daemon stop || true` to release any open file handles the running daemon holds into the npx cache. Tolerant of both running and already-stopped states (`ruflo daemon stop` always exits 0). Without this step, the rm -rf in step 5 can race against an in-flight daemon worker, leaving the next session's `daemon start` to fail silently.
3. **Snapshot** ‚Äî record `ls ~/.npm/_npx > /tmp/ruflo-npx-snapshot-*.list` so you can re-resolve other tools if you depended on a cached version.
4. **Upgrade** ‚Äî `npm install -g ruflo@latest --foreground-scripts` (the `--foreground-scripts` flag works around [npm GH#1825](https://github.com/npm/cli/issues/1825) which silently fails on Node 22/25).
5. **Wipe** ‚Äî `rm -rf ~/.npm/_npx` so the next `npx` resolves cleanly.
6. **Rehydrate** ‚Äî `npx -y ruflo@latest --version` to pull a fresh tree of `@claude-flow/memory@alpha.N`, `agentdb@alpha.M`, etc.
7. **Re-apply patches** ‚Äî runs `fix-ruflo.sh`. Patches 0 + 2 report "already present" on alpha.18; **Step 3b** force-pins the GLOBAL tree's `agentdb` to alpha.10 (the upgrade's `npm i -g ruflo` at step 4 reset it to ^alpha.14). Patch 1 is retired (Tier 7).
8. **Re-activate** (Tier 3, NEW) ‚Äî runs `bin/ruflo-kit init <target> --reactivate`. This re-seats the daemon, swarm, embeddings, neural, hooks, and hive-mind layers against the new code paths. Skips the expensive `ruflo init` / `aqe init` (steps 2/3/4 of the bootstrap) ‚Äî those don't need to re-run after an upgrade because the project files on disk don't change. Without this step, the user is told to "restart Claude Code", which works for the MCP server (lazy-spawned) but NOT for the daemon or swarm state (which need explicit re-seating to bind against the new bundle).

### Before running

- **Close Claude Code first** (the live MCP server holds open handles into the cache).
- Read the project's `docs/UPGRADE-PATH-*.md` if it exists ‚Äî there may be project-specific migration notes (`agentic-qe` version pins, plugin removals, etc.).

### Rollback

If something breaks after the upgrade, the script prints the rollback command in its final summary:

```bash
npm install -g ruflo@<previous-version>
# npx caches auto-rehydrate; nothing else to restore
```

## `ruflo-health.sh` ‚Äî growth-delta monitor (Tier 3, NEW)

The other scripts in this kit answer "did init run?" and "are the patches applied?". This one answers a harder question: **"is the loop actually growing?"** It captures the metrics the statusbar shows (memory entries, SONA patterns/trajectories, MoE accuracy, ReasoningBank patterns, DB rows/sizes, daemon status, hive workers) and compares against the previous run's snapshot to surface growth, flatlines, and regressions.

### What gets tracked

Seven metric groups, all probes are cheap (~1.5s total, no ONNX inference):

| Group | Metrics | Source | Cost |
|---|---|---|---|
| Memory | totalEntries, hnswEntries | `ruflo memory stats` | ~350ms |
| Intelligence (SONA + MoE) | sonaPatterns, sonaTrajectories, sonaQualityPct, moeRoutingAccuracyPct, searchSpeedupLabel | `ruflo hooks intelligence stats` | ~210ms |
| Neural | reasoningBankPatterns | `ruflo neural status` | ~310ms |
| DB rows | swarmMemoryEntries, swarmPatternEmbeddings, swarmVectorIndexes, aqePatterns, aqeTrajectories | `sqlite3 -readonly` probes on `.swarm/memory.db` + `.agentic-qe/memory.db` | ~20ms each |
| DB sizes | swarm (bytes), aqe (bytes) | `stat -f %z` | <5ms |
| Runtime | daemon.running | `ruflo daemon status` (grep RUNNING ‚Äî `$?` is unreliable, see Patch 7's daemon-status notes) | ~200ms |
| Hive | workers | parsed from `.claude-flow/hive-mind/state.json` | ~5ms |

**Intentionally NOT used**: `ruflo embeddings index` (Tier 2 lesson ‚Äî ONNX inference per call is too expensive) and `aqe status` (noisy `[INFO]` chatter from re-init on every call; equivalent info comes from the sqlite probes).

### Files written

- `.claude-flow/data/health-last.json` ‚Äî most recent snapshot (overwritten each run unless `--dry-run`). Schema version 1; future revisions auto-detect version mismatch and treat as a fresh baseline rather than producing nonsense diffs.
- `.claude-flow/data/health-history.jsonl` ‚Äî append-only JSONL history. One line per run. Useful for plotting growth-over-time with a one-liner like `jq -s 'map({iso, sonaPatterns: .metrics.intelligence.sonaPatterns})' .claude-flow/data/health-history.jsonl`.

### Output format

Color-coded delta lines per metric:
- `‚Üë` brightGreen ‚Äî growth (`prev ‚Üí curr (+delta)`)
- `‚Üì` brightRed ‚Äî regression (only triggers exit-1 on monotonic metrics; DB sizes can shrink on cleanup without warning)
- `‚úì` dim ‚Äî stable AND fresh (<24h since baseline)
- `!` brightYellow ‚Äî flatlined (no change AND >24h since baseline AND metric is in the growth class) ‚Äî likely daemon died or hook misfiring
- `‚îÄ` dim ‚Äî label-only metric (e.g. `searchSpeedupLabel: 150x` is a static upstream constant, not a measurement ‚Äî see Tier 2 item 6 lesson)

Ends with a Summary line (`N grew  M stable  K warned  J regressed`) and a Verdict (`‚úì healthy growth ‚Äî learning loop active` | `‚úì all stable (no growth, no regression)` | `! partial ‚Äî some metrics flatlined, check daemon` | `‚úó regression detected ‚Äî investigate`).

### CI-friendly exit code contract

- **exit 0** ‚Äî healthy growth, all stable, or partial (with warnings).
- **exit 1** ‚Äî at least one growth-class metric regressed. Independently verified by injecting `sonaPatterns=999` into the snapshot and confirming the script exits 1. The contract is: if `EXIT=1`, something monotonic went backwards and a human should look.
- **`set -o pipefail` caveat for CI authors**: if you pipe the output through `tail` / `grep` / etc. for log truncation, use `set -o pipefail` (or capture `${PIPESTATUS[0]}`) ‚Äî otherwise the exit code reflects the pipeline's last command (likely 0), masking the regression signal.

### CLI

```bash
bin/ruflo-kit health <target>                # capture current metrics, diff against last snapshot, then update the snapshot
bin/ruflo-kit health <target> --reset        # force re-baseline (overwrite snapshot ‚Äî use after a manual DB wipe)
bin/ruflo-kit health <target> --dry-run      # diff against last snapshot but do NOT update it ‚Äî for passive monitoring in cron/CI
bin/ruflo-kit health <target> --json         # emit raw current snapshot JSON only (suppresses the rendered diff)
bin/ruflo-kit health <target> -h             # help text
```

Auto-detect first-run: if `.claude-flow/data/health-last.json` doesn't exist or has a version mismatch, the script writes a baseline + exits 0 with `baseline established ‚Äî re-run later`. No separate `--init` flag needed.

### Typical usage

- **Per-session at the end**: `bin/ruflo-kit health <target>` to confirm the session's agent activity produced measurable learning growth. Pairs well with `bin/ruflo-kit session <target>` at the start.
- **Cron / CI gating**: `set -o pipefail; bin/ruflo-kit health <target> --dry-run` ‚Äî fails the build if any growth-class metric regressed, doesn't mutate state. Add to `.github/workflows/` or a cron entry that emails on non-zero exit.
- **Post-upgrade verification**: `bin/ruflo-kit upgrade <target> && bin/ruflo-kit health <target> --reset` ‚Äî re-baseline after a major upgrade so post-upgrade growth deltas are measured against the new code's behavior, not the old.

## Local patches (divergence from upstream gist)

The copies of these scripts in `lib/` have been patched locally to fix seventeen issues present in the upstream gist as of 2026-05-27 ‚Äî three in bash escaping, one in the hardcoded version, one in the memory-seed UNIQUE constraint handling, one in the AQE init flow that left projects in a half-initialized state, two structural gaps (missing activation table, missing daemon auto-start) that left the self-learning loop dormant after bootstrap, two statusline rendering bugs (wrong MCP scope filter, file-based vector count instead of sqlite probe) that surfaced after Tier 1 activated the full stack, one defense-in-depth gap (no `node --check` guard on `fix-ruflo.sh` step 11's four `node -e` patching blocks), one statusline accuracy bug (`Sub N` subagent counter stuck at 0 due to two compounding bugs in the file-scan fallback), three statusline-chip dishonesty bugs uncovered by the Tier 4 audit (Vectors undercount, `üíæ MB` chip measuring the renderer's own V8 heap, `üßÝ %` chip derived from a SQLite page-count heuristic), one statusline-domain bug uncovered by the Tier 5 audit (the AgentDB chip's Vectors/Size counters mixed ruflo + AQE state, and the AQE footer surfaced no vector or DB-size signal at all), and five Tier 6 operational + diagnostic gaps (missing agentdb MCP server registration; absent `--force` cleanup of the on-disk hnsw.index sidecar + no post-build invariant check; transient orphan-agentdb warning that didn't explain the upstream re-hoist cycle; missing ONNX-model cache preservation across npx wipes; no visibility into whether HNSW search is using the fast NAPI binary or the silent WASM fallback). Re-downloading from the gist will reintroduce these issues.

1. **`grep -c ‚Ä¶ 2>/dev/null || echo 0`** (fix-ruflo.sh:786,789,802 ; ruflo-session-init.sh:35,47,121‚Äì124) ‚Äî `grep -c` prints `0` on no matches **and** exits non-zero, so `|| echo 0` doubled the output to `"0\n0"`, which then broke `[[ "$VAR" -gt 0 ]]` arithmetic with `syntax error in expression`. Replaced with `|| true` and a `${VAR:-0}` fallback line.
2. **`'\\$1'` inside a `node -e "‚Ä¶"` double-quoted string** (fix-ruflo.sh:945) ‚Äî bash tried to expand `$1` (the script's first positional arg); under `set -u` this aborted the step with `$1: unbound variable`. Replaced with `'\$1'` so bash forwards a literal `$1` to JS, where it becomes the regex backreference.
3. **Slash regex corrupted by bash + JS template literal double-stripping** (fix-ruflo.sh:1083, statusline patch step 11d) ‚Äî source had `CWD.replace(/\\//g, '-')`. Bash collapsed `\\` ‚Üí `\`, then the surrounding JS template literal dropped the `\` before `/` (because `/` isn't a JS string escape), so the file got written as `CWD.replace(///g, '-')`. The `//` then parsed as a single-line comment, breaking the file with `SyntaxError: Unexpected token 'const'` on the next line ‚Äî the rich statusline died and Claude Code silently fell back to the stub. Source now uses `\\\\//g` (four backslashes) so the chain becomes `\\\\` ‚Üí `\\` (bash) ‚Üí `\/` (template literal) ‚Üí final JS `/\//g`. `fix-statusbar.sh` also gained an auto-repair branch (step 1d) that detects the corrupt pattern and rewrites it, so users with the broken file from a prior gist version heal automatically on the next session-init.
4. **Hardcoded ruflo version label** (fix-statusbar.sh) ‚Äî `‚ñä RuFlo V3.10` was a literal in the patcher, and the underlying `statusline.cjs` had a stale `let pkgVersion = '3.6';` runtime fallback. Both now derive from `ruflo --version` (or `npx -y ruflo@latest --version` as a fallback) at script-run time. Two values are exposed: `RUFLO_VERSION_LABEL` (e.g. `V3.10`, for the legacy literal-header branch) and `RUFLO_VERSION_FULL` (e.g. `3.10.3`, for the runtime fallback). Both are inserted only if they differ from what's already on disk, keeping the patch idempotent.
5. **Memory-seed `UNIQUE` constraint on re-run** (init-ruflo-aqe-agentdb.sh, step 7) ‚Äî the original `ruflo memory store -k init ‚Ä¶` would always fail on re-runs because ruflo's SQLite-backed memory enforces `UNIQUE(namespace, key)`, and the failure mode wasn't transient so no retry helped. Step 7 now does a `ruflo memory retrieve -k init` pre-check first; if the key exists, it's counted as a skipped step. If missing, store runs under an exponential-backoff retry (1s / 2s / 4s / 8s, 5 attempts) with a race-check between attempts that treats a concurrent insert as success.
6. **AQE half-init detection and repair** (init-ruflo-aqe-agentdb.sh, step 4) ‚Äî the original script ran `npx -y agentic-qe@latest init --memory hybrid`, which only initializes the AQE SDK (`.agentic-qe/memory.db`, SONA engine, DreamScheduler, Queen Coordinator) but **does not** copy the project-level integration (~115 `.claude/skills/`, AQE agents, `aqe-*.md` commands, the `## Agentic QE v3` section in `CLAUDE.md`). The skip-check then saw `.agentic-qe/memory.db` and concluded "already initialized", silently leaving the project in a half-init state where AQE's MCP tools work but its hooks, skills, and policy rules don't load into Claude Code.
   - **Fix part A** ‚Äî switched to `aqe init --auto` (the canonical first-time flag documented in the [agentic-qe README](https://github.com/proffesor-for-testing/agentic-qe#quick-start)), which performs both SDK init and project integration in one shot.
   - **Fix part B** ‚Äî prereq check (step 1) now detects globally-installed `aqe` (e.g. `aqe 3.10.1`) and prefers it over `npx -y agentic-qe@latest`. The npx path resolves a fresh package per run on machines without `aqe` on `PATH` ‚Äî slower and a frequent source of stale-cache drift.
   - **Fix part C** ‚Äî skip-check now requires **both** `.agentic-qe/memory.db` **and** an AQE-only marker (`.claude/skills/agentic-quality-engineering`, present only after `aqe init --auto`). If the DB exists but the marker is missing, the script identifies it as a half-init and runs `aqe init --auto --upgrade` automatically to complete it.
   - **Fix part D** ‚Äî added a 9th verification check (step 8) for the same marker, so any future half-init from an external tool surfaces immediately rather than hiding behind the 8-check green.

   This was the bug that motivated the cross-check: on this project, the original `--memory hybrid` invocation produced 30 skills (ruflo only) and 6 agents. After switching to `aqe init --auto`, the project gained **+84 skills (now 114 total)**, 13 of them AQE-specific (`qe-test-generation`, `qe-coverage-analysis`, `qe-quality-assessment`, `qe-chaos-resilience`, `agentic-quality-engineering`, etc.), plus the AQE policy section in `CLAUDE.md`. ## Tier 10 ‚Äî Objective reward oracle + Router-B production closure (this session)

The self-improvement work prior to this tier had a CLOSED-LOOP *mechanism* on the AQE router but two unresolved weaknesses: (a) the reward fed to the trainers was a constant (0.8) or a gameable prose-sentiment heuristic, and (b) Router B (`ruflo hooks route`) had no production training store the CLI could write, so its learning was inert on real turns. Channel chosen over npm/brew: a git-tracked patch kit that pins to upstream ruflo versions wants to track `main` directly, not carry publish/version-bump ceremony. **(c) New dispatcher verbs:** `version` (kit git sha + detected global ruflo/agentic-qe versions) and `self-update` (`git pull --ff-only` + the Patch-44 "re-run fix-ruflo after a ruflo bump" reminder). **(d)** Native builds stay **bundled + darwin-arm64** (1.3M; install already platform-gated, skips cleanly elsewhere) ‚Äî README states the scope honestly. **(e)** Added a single CI smoke (`.github/workflows/shellcheck.yml`: `bash -n` + shellcheck over `install.sh`/`bin`/`lib` + a symlink-resolution assertion) ‚Äî the repo's first CI, cheap insurance now that it's `curl|bash`-distributed. Clean break: no back-compat shims; `bin/ruflo-kit` and global `ruflo-kit` are the same entrypoint.

**(46) Post-task hook arg fix ‚Äî Task trajectories were unpromotable (`fix-aqe.sh`, `AQE-POSTTASK-ARGS-V1`).** Investigating "only 2 patterns in the statusbar" (despite 300+ trajectories) traced to the stock post-task hook that `aqe init` generates: `npx agentic-qe hooks post-task --task-id "$TOOL_RESULT_agent_id" --success --json`. Two defects, both proven live: **(1)** the CLI declares `--success <bool>` (value *required*); passing bare `--success --json` makes commander consume `"--json"` as the value, so `e.success !== "true"` ‚Üí **every** trajectory + captured_experience writes `success=0` (verified 305√ó0 / 1√ó1), AND `--json` is swallowed (the hook's JSON contract is void). **(2)** no `--agent` is passed ‚Üí `agent='unknown'` on every row. Both starve distillation: `qe_trajectories` are keyed by agent/domain/success for promotion, so `unknown/general/0` rows can never become patterns ‚Äî the only 2 patterns came from the separate cli-hook `captured_experiences` path. **Fix** (in the existing `^(Task|Agent)$` PostToolUse group, where the env vars are real): rewrite to `--task-id "$TOOL_RESULT_agent_id" --agent "$TOOL_INPUT_subagent_type" --success true --description "$TOOL_INPUT_prompt" --json`. Idempotent (skips if already `--success true`), rewrites only the agentic-qe post-task command, survives an `aqe init` regen (the stock generator keeps emitting the broken form). **Verified with a 2√ó loop:** firing the corrected hook with distinct agents wrote `success=1` + real `agent` (`qe-test-architect`, `qe-security-scanner`) to BOTH `qe_trajectories` and `captured_experiences`; the `--json` output is now valid. **Known limitation (documented, out of scope):** post-task has **no `--domain` flag** and the writer (`Z()`) never sets domain, so the trajectory's `domain` column stays `'general'` regardless of `--description` (which only feeds upstream's separate pattern-bridge). Real per-domain trajectories would need a dist patch to the writer ‚Äî deferred. This removes the promotion *blocker*; pattern growth still depends on domain diversity (distillation keys on `domain √ó month`). **Process note:** during prototyping I corrupted the live `.agentic-qe/memory.db` by `cp`-ing a backup over a live WAL-mode DB (active daemon + MCP writers) ‚Äî recovered fully via `sqlite3 .recover` (`integrity_check ok`, 0 rows lost) + WAL-mode restore + Claude Code restart to drop the stale MCP page-cache. **Rule: never `cp` over a live SQLite DB; use `sqlite3 .backup` / surgical `DELETE`, or stop all writers first.** **Reported upstream:** [proffesor-for-testing/agentic-qe#508](https://github.com/proffesor-for-testing/agentic-qe/issues/508) (the generated-hook `--success`/`--agent` defect, with repro + fix) ‚Äî when fixed upstream, `AQE-POSTTASK-ARGS-V1` becomes a no-op (it already skips when the hook is correct).

**(47) Dream-scheduler runaway counter ‚Äî investigated, confirmed cosmetic, NOT dist-patched (kit issue #1).** Kit-repo issue #1 reported the `dream-scheduler:hook-state` counter as a "runaway" (`lastDreamTime:null`, `experienceCount` climbing to thousands, firing a dream on every post-task hook) and proposed `AQE-DREAM-STATEFIX` (move the state-write before `engine.dream()` in `checkAndTriggerDream`). **Investigation against live `agentic-qe@3.10.1` found the diagnosis fits a PRE-ADR-094 layout, not the shipped one.** In 3.10.1: the post-task hook calls ONLY `incrementDreamExperience` (bumps the counter) ‚Äî it does **not** trigger dreams; `checkAndTriggerDream` is *intentionally NOT re-exported* to the hook path (`hooks-shared.js:352`, "ADR-094"). Dreams run in the **kernel-side `DreamScheduler`** (`dist/learning/dream/dream-scheduler.js`, long-lived daemon) on its own timer; it keeps its own in-memory `lastDreamTime` and **never writes the `DREAM_STATE_KEY` kv row** the SessionStart readout reads. So the climbing counter is a pure **writer/reader split** (hook writes, daemon dreams, nothing reconciles) ‚Äî **cosmetic, NOT a runaway** (verified: dreams ran fine, 262 completed / 198 failed on their own cadence regardless of the counter). The proposed STATEFIX, while a correct/harmless hardening, sits on the kernel/dream-engine/mincut path ‚Äî NOT the post-task hook ‚Äî so it would not fix the readout. **Decision (extends 43(c)): stay cosmetic ‚Äî do NOT dist-patch.** A daemon-side `DreamScheduler`‚Üí`DREAM_STATE_KEY` write is the real 3.10.1 fix but is exactly the version-fragile reader-reconciliation 43(c) chose to avoid, for a value the kit's statusbar already bypasses (it reads `dream_cycles`, the truth source). **One-time heal** when the readout bothers you (re-climbs after ‚Äî that IS the cosmetic nature): `UPDATE kv_store SET value = json_set(value, '$.lastDreamTime', (SELECT MAX(end_time) FROM dream_cycles WHERE status='completed'), '$.experienceCount', 0, '$.totalDreamsThisSession', 0) WHERE key='dream-scheduler:hook-state';` (back up via `sqlite3 .backup` first ‚Äî see Patch 46's WAL rule). Issue #1 to be updated with this reconciliation + re-titled toward the daemon `DreamScheduler` non-reconciliation; reported upstream: [proffesor-for-testing/agentic-qe#509](https://github.com/proffesor-for-testing/agentic-qe/issues/509) (kernel DreamScheduler should reconcile the hook-state kv row).

**(48) Native-binary check was darwin-only ‚Üí false "native not found" on linux-arm64 (`session-init.sh` + `health.sh`).** A `ruflo-kit session` on a DGX Spark (Linux/ARM64) warned "Attention native not found" at step 7/7 even though `attention.linux-arm64-gnu.node` was present in the global ruflo install. Two macOS-baked assumptions: **(1)** step 7 mapped `process.arch` directly (`arm64 ‚Üí darwin-arm64`), ignoring `process.platform` ‚Äî so on Linux it searched for a macOS filename that can't exist; **(2)** both `session-init.sh` step 7 and `health.sh`'s `detect_hnsw_backend` searched only `~/.npm/_npx` (+ `~/node_modules`), missing the global-ruflo nested `node_modules/@ruvector/...` where `npm i -g ruflo` actually lands the binaries ‚Äî and `health.sh` built `linux-arm64` without the `-gnu` libc suffix @ruvector uses. **Fix:** two shared helpers in `common.sh` ‚Äî `ruvector_platform_tag()` (node `process.platform`+`process.arch`+libc ‚Üí `darwin-arm64` / `darwin-x64` / `linux-arm64-gnu` / `linux-x64-gnu` / `linux-*-musl` on Alpine / `win32-x64-msvc`; no libc suffix on darwin) and `ruvector_search_roots()` (npx cache + `~/node_modules/@ruvector` + `$(npm root -g)/ruflo/node_modules` + `$(npm root -g)/@ruvector`). Both `session-init.sh` step 7 and `health.sh` now use them. Verified: real run on darwin-arm64 still passes all three natives (no regression); a faked-node linux/arm64 probe yields `linux-arm64-gnu` (matching the DGX's actual binary). Warnings now also say "(WASM fallback active)" so a genuine miss reads as degraded-not-broken (attention/SONA/GNN fall back to WASM ‚Äî slower, functional). NOTE: the kit still only *bundles* darwin-arm64 prebuilts (`assets/builds/`, install gated on `uname==Darwin`); linux hosts rely on @ruvector's upstream `linux-*-gnu` optional-deps that `npm i -g ruflo` pulls ‚Äî this patch fixes *detection*, not provisioning.

## Safety notes

- **Dry-run mode**: `fix-ruflo.sh --dry-run` and `upgrade-2026-05-26.sh --dry-run` print every action they would take without modifying anything. Use this first if you're unsure.
- **Idempotency**: the three "fix-*" scripts can be re-run any number of times. They detect already-applied patches and skip them.
- **Logging**: `fix-ruflo.sh` and `upgrade-2026-05-26.sh` tee to `/tmp/fix-ruflo-*.log` and `/tmp/upgrade-ruflo-*.log` respectively. Keep these around for post-mortem if a step fails.
- **Scope**: the scripts touch user-scope locations only. No `sudo`, no system files, no production state. Recordings, project source code, and git state are untouched.
- **Network**: every step that needs the network catches errors and continues. If you're on an air-gapped machine, the `npm view`/`npm install` steps will fail loudly but the remaining diagnostic steps still run.
- **What is NOT modified**: the scripts never touch `.git`, never commit, never push, never re-write production source. They do not touch the project's `package.json` except to **remove** known-conflicting dev deps (and only after detection).

## FAQ

**Q. `agentdb_controllers` still says 15+ controllers disabled after I ran `fix-ruflo.sh` and restarted Claude Code.**
Run `bin/ruflo-kit fix-ruflo <target>` once more and look at Step 3/11 ("Shadow agentdb detection") and Step 10/11's "AgentDB import" output. A new shadow may have appeared (a fresh `npx` of an unrelated tool can re-introduce one). If the output looks clean and the issue persists, wipe the npx cache: `rm -rf ~/.npm/_npx && npx -y ruflo@latest --version && bin/ruflo-kit fix-ruflo <target>`.

**Q. `memory_store` works but `memory_search` returns nothing.**
That usually means the HNSW vector index is out of sync with the SQLite memory table. Confirm `.swarm/hnsw.index` exists and has a recent mtime; if not, store one entry to trigger rebuild: `npx ruflo@latest memory store --key bootstrap --value ok --namespace project`.

**Q. The status line says "‚ñä RuFlo + Agentic QE v3" with no live numbers.**
That's the literal fallback string in `statusLine.command`. Both Node scripts failed. Most often `.agentic-qe/memory.db` is missing ‚Äî run `aqe init --auto` to create it, then `bin/ruflo-kit fix-statusbar <target>` to re-patch.

**Q. `ruflo-session-init.sh` reports "MCP server not running".**
Claude Code launches the MCP server lazily on the first MCP tool call. The warning is informational; once you do anything that needs MCP, the server will spawn.

**Q. Should I commit `.swarm/` and `.claude-flow/` to git?**
The persistent files (`memory.db`, `hnsw.index`, `graph-state.json`, `intelligence-snapshot.json`) are useful **per-developer** memory and can be committed if you want cross-session, cross-machine recall. The transient lock files (`*.lock`, runtime sockets) should be gitignored. See your project's `.gitignore` for the canonical list.

**Q. I'm on Linux / Windows / WSL ‚Äî does this work?**
Linux is fully supported (the script branches on `uname` for the `sed -i` syntax difference). Windows is not supported directly; use WSL.
