### Synthesis

The SOTA state isn't a different install — it's an install that's done **two more things** beyond `ruflo init`:

1. **Activated the learning loop**: `swarm init` → `embeddings init` → `memory init --build-hnsw` → `neural train` → `hooks pretrain` → `daemon start` → `hive-mind init`.
2. **Registered the architecture surface**: `ruflo-ddd` plugin to scaffold bounded contexts, `ruflo-adr` plugin to register ADRs.

Once both run, the agents start producing trajectories, the daemon writes them as patterns, vectors grow, HNSW becomes meaningful, and the intelligence % climbs. The screenshot is a project that's been doing this for ~3.5 hours (`206m2s`) and has cost ~$84 of model tokens — meaning real agent work was happening, not just CLI ceremony.

## 2. Per-script gaps

### 2.1 `init-ruflo-aqe-agentdb.sh` — the largest gap

This is the bootstrap entry point. It currently runs `ruflo init` + `ruflo memory init` + `aqe init --auto` + the two fix scripts. It **does not** run any of the activation steps above. The result is the "all zeros" statusline we currently see.

**Missing activation steps (must be added):**

| # | Command | Skip condition | What it produces |
|---|---|---|---|
| A | `ruflo swarm init --topology hierarchical-mesh --max-agents 15` | `.claude-flow/swarm/swarm-state.json` exists with `agents>=0` and `updatedAt < 30 min` | Coordinator + queue; required for any agent dispatch |
| B | `ruflo embeddings init` | `embeddings_status` JSON shows `provider: Xenova/all-MiniLM-L6-v2` initialised | ONNX embeddings; required before HNSW can index |
| C | `ruflo memory init --build-hnsw` (or `embeddings_rabitq_build` if HNSW isn't a memory-init flag) | `.swarm/hnsw.index` exists and is non-empty | The HNSW index file; flips the "⚡ HNSW 150x" indicator on |
| D | `ruflo neural train` (or `neural pretrain`) | `neural_status` reports active patterns >0 | Seeds the SONA self-learning loop; required for intelligence % to climb |
| E | `ruflo hooks pretrain` | `hooks_intelligence_stats` reports non-zero patterns | Pre-populates the hook pattern store so routing decisions have priors |
| F | `ruflo hive-mind init` | `.claude-flow/hive-mind/state.json` exists | Collective memory backbone; required for cross-session recall |
| G | `ruflo daemon start` | `ruflo daemon status` reports `Status: ● RUNNING` | The 5–12 background workers that actually grow the system between user interactions |
| H | `ruflo memory store -k "session/<datetime>" -v "<git rev>"` (replaces step 7) | Always seed (multiple session markers OK — no UNIQUE collision since key is unique per session) | Cleaner per-session seeding; survives indefinitely without the retry/dedup dance |

**Discovered issue:** the current "AQE init flag" plumbing is correct (`--auto`, global binary preferred, half-init repair) but **no equivalent activation table exists for ruflo**. The bootstrap calls `ruflo init` (which only writes templates) and `ruflo memory init` (which only creates an empty DB). Everything in the table above is missing.

**Discovered optimization:** the existing `--force` flag should also cascade into `daemon stop && daemon start` so a re-run with `--force` re-seats workers (otherwise a stale daemon keeps running with the old config).

**Order matters:** A → B → C → D → E → F → G. `daemon start` must come **after** the data plane (memory + embeddings + HNSW) is up, otherwise workers crash-loop. `neural train` requires embeddings; `hooks pretrain` requires memory. Add a guarded dependency check.

### 2.2 `fix-statusbar.sh` — mostly aligned, two refinements

**Aligned:**
- ✅ Header `RuFlo V<dynamic>.<X.Y.Z>` matches (after our dynamic-version fix)
- ✅ AQE footer present
- ✅ Auto-repair for `///g` corruption is in place

**Gap 1 — stdin field plumbing:** the screenshot shows model name, context %, uptime, cost in the header. The current `statusline.cjs` already grep-references `stdinData.subagents` / `num_subagents`, but **does not** appear to render `model`, `context_used_pct`, `cost_usd`, `session_uptime_ms` even though Claude Code passes them on stdin. Action: add a patch step `1e` that detects whether the header builder consumes these fields and, if not, injects a small renderer block that reads `stdinData.model`, `stdinData.cost_usd`, etc., and appends them to the header line. This is the difference between `▊ RuFlo V3.10.3 ● user │ Claude Code` (current) and `▊ RuFlo V3.10.3 ● user │ Opus 4.7 (1M context) │ ⏱ 206m2s │ ● 69% ctx │ $83.72` (SOTA).

**Gap 2 — HNSW speedup indicator:** statusline currently reads pattern count and shows "📚 42 patterns" on the same line where the SOTA shows `⚡ HNSW 150x`. The data needed is `embeddings_status.hnsw.speedup_factor` (or compute from `vectors_count` and a heuristic). Action: add a patch step that, if `.swarm/hnsw.index` exists and is non-empty, replaces the pattern-count chip with `⚡ HNSW 150x`.

**Aligned but worth noting:** the script is **only triggered after `aqe init`** to repair clobbering. It does NOT trigger on first install. The bootstrap script does call it, so that's covered — but `ruflo-session-init.sh` should also invoke it unconditionally on every session start (already does).

### 2.3 `fix-ruflo.sh` — three optimization opportunities

**Aligned:** the 11 diagnostic steps are sound. The local fixes (slash regex, `\$1`, `grep -c` doubling) are in.

**Optimization 1 — Step 8 is incomplete (ghost-state).** Currently resets `agents.json`, `hive-mind/state.json`, and `swarm-state.json` based on size/age heuristics. Missing: **`.claude-flow/data/intelligence-snapshot.json`** and **`.claude-flow/learning/`** — both can accumulate stale entries across upgrades. Add a similar age-bounded reset.

**Optimization 2 — Statusline patch 11d (the one we just fixed) sets the project-key regex but never validates that the resulting regex compiles.** A simple post-write `node -e "new RegExp(...)"` would have caught the `///g` corruption on the run that introduced it, instead of silently writing a broken file. Add a post-patch syntax check: `node --check .claude/helpers/statusline.cjs` after every modification, and abort + restore the backup if it fails.

**Optimization 3 — No bridge between `fix-ruflo.sh` and the activation table.** The script ends its summary with "Restart Claude Code to apply changes." But there's no nudge to run the activation commands. A new final step (12/12) could check `daemon status` + `swarm_status` + `embeddings_status` and warn if any are inactive, with a one-line hint to re-run the bootstrap.

### 2.4 `ruflo-session-init.sh` — under-uses the activation table

This is the "every session" entry point. It currently checks:
- 5 file presences (`memory.db`, `hnsw.index`, `graph-state.json`, `intelligence-snapshot.json`)
- AgentDB controller patches
- RuVector native binaries

**Missing checks (all from the activation table):**
- Is the daemon running? If not, suggest `ruflo daemon start` or auto-start it.
- Is the swarm initialised (recent `swarm-state.json`)? If not, run `ruflo swarm init`.
- Is hive-mind initialised? If not, run `ruflo hive-mind init`.
- Has neural-training ever run? If not, suggest one-time `ruflo neural train`.
- Are pattern-store + intelligence-snapshot files growing across sessions, or flatlining? A delta check (e.g., compare size to last session's snapshot) surfaces "the loop has stopped working" — the case where statusline numbers stop climbing.

**Optimization:** session-init is the right place to *re-start* the daemon if it died between sessions. Add a `daemon status; if STOPPED then daemon start` block. This alone closes the gap that currently shows "Status: STOPPED" on our project right now.

### 2.5 `upgrade-2026-05-26.sh` — needs a re-activation tail

The upgrade script wipes `~/.npm/_npx` and re-resolves ruflo, then re-runs `fix-ruflo.sh`. **Missing:** after a major upgrade, the daemon must be stopped before the wipe and re-started after; otherwise the running daemon holds open handles into the cache and the next session's daemon-start fails silently.

Order should be:
1. `ruflo daemon stop` (currently absent)
2. wipe npx
3. re-resolve
4. `fix-ruflo.sh`
5. **NEW** — re-run the activation table from `init-ruflo-aqe-agentdb.sh` (or invoke it directly with a `--reactivate-only` flag)
6. `ruflo daemon start`

Currently the user is told "restart Claude Code"; that works because Claude Code lazy-starts the MCP server, but the daemon and the swarm don't auto-restart, so the user silently loses self-learning until they manually `daemon start`.

## 3. Cross-cutting opportunities

### 3.1 Unified "activation table" module

All five scripts ultimately need to run subsets of the same activation table. Extract it into a sourced helper at `lib/ruflo-activate.sh` exposing:

```bash
ruflo_activate_swarm    [--force]
ruflo_activate_memory   [--force]      # embeddings + memory + HNSW
ruflo_activate_neural   [--force]
ruflo_activate_hooks    [--force]
ruflo_activate_hivemind [--force]
ruflo_activate_daemon   [--force]
ruflo_activate_all      [--force]      # runs all of the above in order
```

Each function is idempotent (checks its own skip condition), supports `--force`, emits standard pass/fail/warn lines, and returns 0/1. Then:
- `init-ruflo-aqe-agentdb.sh` calls `ruflo_activate_all` after step 6
- `ruflo-session-init.sh` calls `ruflo_activate_daemon` (only — the rest survive between sessions)
- `upgrade-2026-05-26.sh` calls `ruflo_activate_all --force` after the wipe
- `fix-ruflo.sh` step 12 (new) just reports the status of each function without running them

This eliminates the current bug where the four scripts have diverging ideas of "what done looks like".

### 3.2 Verification growth check

The screenshot's most informative metric is **growth over time**: Vectors 1429, DB 121MB, intelligence 71%. A new script `lib/health.sh` could:
- Read the previous snapshot from `.claude-flow/health-last.json`
- Compare against current `memory stats` / `neural status` / `hooks intelligence stats`
- Print "growth since last check" deltas
- Warn if any metric flatlined > 24h (likely daemon died or hook misfiring)

This is the difference between "did init run?" (current scripts) and "is the loop actually working?" (the question the screenshot answers).

### 3.3 Header-line stdin renderer

The model/context/uptime/cost block is the most visible gap. Worth a dedicated `fix-statusbar.sh` step (1e) that:

1. Detects whether `statusline.cjs` already reads `process.argv[2]` or `JSON.parse(fs.readFileSync(0))` for stdin data.
2. If yes, verifies it renders `model`, `cost`, `context_used_pct`, `session_uptime_ms`.
3. If no, injects a small block that does, anchored on the existing `header = ...` line.

Without this, even a perfectly-activated project will look "incomplete" because the chrome doesn't surface the live numbers Claude Code is already sending.

### 3.4 Statusline structural validation

Append to every script that writes `statusline.cjs`:

```bash
if ! node --check "$STATUSLINE_FILE" 2>/dev/null; then
  warn "statusline.cjs failed syntax check — restoring backup"
  mv "$STATUSLINE_FILE.bak" "$STATUSLINE_FILE"
fi
```

This is a cheap defense-in-depth against the `///g`-style template-literal escape bugs we already hit once.

### 3.5 `--reactivate` flag on the bootstrap

A common workflow is "I bootstrapped a while ago but something stopped working." Currently the answer is `--force`, which is heavy-handed. Add `--reactivate`:

- Skip steps 2–4 (init / memory init / aqe init)
- Run only steps 5, 6, and the new activation table
- Re-seat the daemon, verify all activation checks

## 5. Verifiable acceptance criteria

After implementing Tier 1 + Tier 2, a fresh bootstrap on a new project should yield (within ~2 minutes of `ruflo daemon start` + a few minutes of agent activity):

- `ruflo daemon status` → `Status: ● RUNNING`
- `.swarm/hnsw.index` → exists, ≥4 KB
- `ruflo memory stats` → Total Entries ≥ 5 (init + swarm session + daemon worker entries)
- `ruflo neural patterns` → ≥1 pattern
- `ruflo hooks intelligence stats` → enabled, ≥1 trajectory
- Statusline header → renders model + ctx % + cost (assuming Claude Code provides them)
- Statusline AgentDB row → Vectors ≥1, Size > 200 KB

A working ruflo loop will then organically grow these numbers toward the screenshot's `Vectors 1429 / Size 121MB / 🧠 71%` over normal use.

## 7. Open questions

1. **`ruflo memory init --build-hnsw`** — verify the flag name. The current `memory init` already reports `HNSW Indexing: ✓ Enabled`, but `.swarm/hnsw.index` doesn't exist until at least one vector is stored. Possibly the index is built lazily on first store; if so the activation table just needs a "store one trigger entry" step, not a separate flag.

3. **`hooks pretrain` cost** — pretraining may take minutes on a large project. Add a `--skip-pretrain` flag for fast bootstraps.
4. **Daemon and Claude Code MCP server lifecycle** — confirm they don't compete for the same lock. If they do, the activation table needs explicit ordering relative to MCP.
